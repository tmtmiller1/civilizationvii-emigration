# Emigration — Civ VII 1.4.1 deep pass

Civ VII **1.4.1** ("Test of Time" line) substantially reworked **happiness, governments, and
celebrations**. This plan brings the migration model into alignment with those systems. It is the
authoritative spec for the change; implement it as written.

## 0. Compatibility verdict (verified against the installed 1.4.1.28 game data)

The mod does **not** break on 1.4.1. Every live read the simulation depends on still resolves in
1.4.1's own UI code, and every base-game identifier the mod's policy XML references still exists:

- `city.Happiness.netHappinessPerTurn` and `city.Happiness.hasUnrest` — still valid city-level reads
  (confirmed in `base-standard/ui/.../city-banners.js` and elsewhere).
- `city.Yields.getNetYield`, `city.isBeingRazed`, `addRuralPopulation`, `UNIT_MIGRANT`,
  `Culture.isTraditionActive` — all present.
- All 12 `NODE_CIVIC_*` civic-tree foreign keys the three `data/emigration-policies-*.xml` files hang
  off still exist; `POLICY_CULTURE_SLOT` and `KIND_TRADITION` intact. The policy data still loads.
- Government type IDs were **not** renamed (`GOVERNMENT_CLASSICAL_REPUBLIC`, `_DESPOTISM`,
  `_OLIGARCHY`, `_FEUDAL_MONARCHY`, `_PLUTOCRACY`, `_THEOCRACY`, `_AUTHORITARIANISM`, …).

So this is an **additive modeling + recalibration** pass, not a repair job. Nothing below removes or
renames an existing read, tunable, or persisted key. Every new behavior is behind a flag and
defaults to a conservative, bounded contribution.

## 1. What 1.4.1 changed (the parts that drive migration)

1. **Happiness is now a formal 5-stage ordinal per settlement**, defined in `GameInfo.HappinessStages`
   with `StageMinThreshold`/`StageMaxThreshold` (thresholds scale by Age):
   `HAPPINESS_STAGE_ANGRY < HAPPINESS_STAGE_UNHAPPY < HAPPINESS_STAGE_HAPPY < HAPPINESS_STAGE_JOYOUS < HAPPINESS_STAGE_ECSTATIC`.
   The base game buckets a city's `netHappinessPerTurn` by `h >= min && h <= max`.
2. **The unhappiness yield penalty roughly doubled (~−2% → −5% per point, max −80%, Food exempt).**
   This already propagates into the model *indirectly*: an unhappy city's suppressed yields are read
   through `getNetYield`, so `productiveness` already drops harder. We must avoid double-counting it.
3. **Governments carry happiness-stage-keyed passives + two Golden Ages + persistent Government
   Traditions.** Most of a government's effect reaches the model through the happiness/yield it
   produces (already read). The residual, not-yet-modeled signal is the government's *flavor lean*.
4. **Celebrations (engine term: Golden Age) are now scarcer (~6 turns, escalating threshold) and feed
   Tourism / Cultural Victory.** A civ in a Golden Age is a genuine, discrete attractor that the raw
   happiness number does not fully capture. Live reads: `player.Happiness.isInGoldenAge`,
   `.goldenAgeTurnsLeft`, `.nextGoldenAgeThreshold`.
5. **War weariness** is an empire-wide unhappiness source distinct from in-border combat. Live read:
   `player.Happiness.hasWarWeariness`. The mod's violence model only sees fighting *inside a city's
   borders*; empire-wide war fatigue is currently invisible to it.

## 2. Design

Keep the model shape `Prosperity = base × (1 + Σ situational%/100)`. Add three new, bounded inputs;
do not perturb the existing happiness/yield calibration (it auto-absorbs the −5% change through
yields, see §1.2).

### 2a. New reads — `ui/emigration-polity.js` (new module)

- `cityHappinessStage(city) → -2..+2` — bucket `city.Happiness.netHappinessPerTurn` against a cached
  parse of `GameInfo.HappinessStages` (ANGRY −2, UNHAPPY −1, HAPPY 0, JOYOUS +1, ECSTATIC +2). Fully
  defensive (returns 0 on any failure). Mirrors `city-banners.js` bucketing exactly.
- `readPolity(owner) → { government, celebrating, goldenAgeTurnsLeft, warWeary }` — per-civ, memoized
  per pass. Reads `Culture.getGovernmentType()` → `GameInfo.Governments.lookup().GovernmentType`,
  `Happiness.isInGoldenAge`, `.goldenAgeTurnsLeft`, `.hasWarWeariness`. Neutral defaults on failure.
- `resetPolityCache()` — clear the per-pass memo (called once at the top of `collectCitySignals`).
- `governmentLean(governmentType) → small signed number` — a conservative, bounded flavor table for
  the known 1.4.1 governments (happy/celebration-leaning governments slightly more attractive). Kept
  small precisely because the bulk of a government's effect already flows through happiness/yields;
  this is a tie-breaker/flavor term, not a primary driver. Unknown government → 0.

### 2b. Signal — `ui/emigration-cities.js`

`CitySignal` gains `stage` (per-city ordinal) and `polity` (the per-owner object). `buildSignal`
reads both; `collectCitySignals` calls `resetPolityCache()` first. Typedef updated.

### 2c. Scoring — `ui/emigration-prosperity.js`

- `baseScore` gains a bounded additive `polityBonus(s, tune)` (added in BOTH the legacy-linear and
  shaped branches), scaled by `tune.happinessPull` so the per-civ table still modulates it:
  - `happinessStageWeight × stage` — a robust, magnitude-insensitive happiness response that
    complements the existing field-relative term and gives the harsher 1.4.1 happiness swings a
    bounded, ordinal voice.
  - `+ celebrationPull` while `polity.celebrating`.
  - `+ clamp(governmentWeight × governmentLean(govt), ±governmentLeanCap)`.
- `situationalPercent` gains `if (polity.warWeary) pct += warWearinessModifier` — a modest empire-wide
  push. Because it lives in the situational factor it composes with (and is dominated by) in-border
  violence, so it never double-counts a city already under siege.

### 2d. Causes / persistence — unchanged

Government, celebration, and war weariness fold into the existing `prosperity`/`unhappiness`
attribution. No new `MigrationCause` value (the cause set is additive-only and persisted; a new value
would require a load-time alias for no real gain). No persisted-key changes.

### 2e. Config — `ui/emigration-config.js` + `emigration-config-types.js`

New, flag-gated, conservative tunables (all reversible; `polityModelEnabled: false` restores exact
pre-1.4.1 behavior):

```
polityModelEnabled: true,
happinessStageWeight: 4,    // bounded pull per stage step (ANGRY −2 … ECSTATIC +2)
celebrationPull: 6,         // attractiveness while a civ is in a Golden Age (celebration)
governmentWeight: 2,        // scales the per-government flavor lean
governmentLeanCap: 3,       // clamp on the government lean term
warWearinessModifier: -12,  // empire-wide situational push (%) for a war-weary civ
```

## 3. Tests (`tests/polity.mjs`, plus `tests/prosperity.mjs` additions)

- `cityHappinessStage` buckets correctly across all five stages (stub `GameInfo.HappinessStages` +
  a city with a given `netHappinessPerTurn`), and degrades to 0 with no game globals.
- `readPolity` extracts government/celebration/war-weariness and memoizes per pass; neutral on
  failure.
- `prosperity`: with `polityModelEnabled`, celebration adds the expected pull, the stage term scales
  with `stage`, the government lean clamps to `±governmentLeanCap`, and `warWeary` applies the
  situational percent. With the flag off, scores match the pre-pass baseline exactly.
- Register `test:polity` in `package.json` (`test:js`, `verify`).

## 3a. Balance pass (happiness/economy recalibration)

The −5%/point happiness change prompted a full balance review (three analysis harnesses under
`scripts/`, all driving the real `prosperity()` off-engine via the test loader):

- **`happiness-balance.mjs`** — measured that, in the *original* tuning, the −5% change was nearly
  inert (it moved the content→unhappy pull gradient by ~1%), because the shaped happiness model so
  dominated the score that the yield channel (where −5% lives) was drowned out.
- **`calibration-sweep.mjs`** — swept the five interacting balance parameters (`happyFloor`,
  `happyAmp`, `happyRepulsion`, `happyScale`, yield-weight gain) over a happiness×economy world,
  scored against a player-experience rubric: monotonic in both happiness and economy, economy a
  strong-but-minority share of the signal (~target 0.4), happiness multiplier not clamp-saturated,
  the −5% change visible, bounded gradient, and misery still biting. Current defaults scored 0.38
  (economy only ~10% of the signal, multiplier saturated 43% of the time). The chosen calibration
  (`happyFloor 4`, `happyAmp 0.2`, `happyRepulsion 1.8`, yield factors ×2.5) is the least-disruptive
  point passing all guards: economy ~28% of the signal, fully de-saturated, −5% 8× more visible,
  overall scale held so downstream friction/pacing constants stay valid.
- **`snowball-stress.mjs`** — confirmed the rebalance does NOT worsen runaway leaders. The dominance
  ceiling (popRatio at which a leader stops gaining cross-civ migrants) is equal-or-lower than the old
  defaults for happy/rich/rich+happy leaders, because de-saturating lifts the field and shrinks the
  prosperity gap that feeds a leader; an economy-only leader's gap stays under the cross-civ friction
  so it can't snowball through migration at all.

Final balance sign-off still wants in-game observation; these harnesses bound the risk and are
re-runnable when re-tuning. The polity tunables (§2e) were sized against the post-rebalance scale.
